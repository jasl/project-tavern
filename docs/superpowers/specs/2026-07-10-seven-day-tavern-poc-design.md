# 七日酒馆经营 PoC 设计规格

日期：2026-07-10
状态：历史设计评审摘要；技术架构已由 `2026-07-10-react-game-harness-design.md` 取代
对应里程碑：第一个可玩的数值灰盒

> 本文保留最初选择七日 PoC 的推理。七日玩法范围仍由 `docs/poc/` 管理；一次性 Web 代码、纯数据内容、旧存档和旧 UI 约定不再作为实现依据。

## 1. 设计结论

采用一个独立的 React + TypeScript 模拟实验室，而不是直接进入 Unity。这个原型的实现可以丢弃，但规则、稳定 ID、内容数据、确定性随机向量、存档 fixture 和验收场景应能被未来 C# 实现复用。

首轮只实现一个非正史七日周循环，以税负形成最低经营压力，以一个关系/调查互斥窗口验证路线取舍，以营业委托验证自动化的价值。

## 2. 选择这个方案的原因

考虑过三条路线：

1. **直接 Unity**：更接近最终运行时，但会让 UI、场景、资源导入和引擎学习干扰数值问题；
2. **表格/命令行模拟**：最快验证公式，但难以验证信息层级、操作疲劳和关系事件的交互；
3. **React + 纯 TypeScript 模拟核心**：仍然便于快速改数值，同时能做真实可点击流程、预测、拆账、存档和调试。

选择第三种。它比电子表格更接近真实游玩，又不会把正式 Unity 架构过早锁死。

## 3. 玩家循环

1. 查看本周义务、今天的客流预测和可选机会；
2. 在上午/下午进行采购、备菜、休息或事件行动；
3. 选择最多两道菜、计划份数与营业方式；
4. 确认成本与风险预测；
5. 结算需求、容量、库存、收入、工资、状态和关系；
6. 查看逐项账本并进入下一天；
7. 周日缴税并得到经营、关系、调查三个独立结果。

## 4. 关键取舍

- 亲自营业现金和默契最好，但持续消耗玩家体力与晚上行动点；
- 部分委托释放一个晚上行动点，并承担少量工资与容量损失；
- 完全委托释放整个晚上，但现金缓冲很薄；
- 计划停业保护人气和体力，却必须靠其他营业日补足税负；
- 周五关系与调查互斥；冒险还会消耗体力，从而影响当晚经营；
- 冷藏设施降低库存风险，舒适床铺提高可投入劳动，两者只可选一个。

## 5. 系统边界

### 5.1 模拟核心

纯 TypeScript 核心拥有：时段、行动点、体力、现金、库存、菜谱、需求、营业、关系简化状态、Aura、事件、确定性随机和存档数据。

所有修改通过命令执行：

`executeCommand(state, command, definitions) -> { nextState, facts }`

失败命令返回结构化原因并保持原状态。时间只通过一个入口推进；结算以完整事务生成新状态，不能部分提交。

### 5.2 内容

只读 `GameDefinitions` 保存原料、菜谱、客群、事件和设施。内容使用稳定 ID 与有限的条件/效果联合类型，禁止任意表达式、脚本字符串和运行时 TypeScript。

### 5.3 React UI

React 只投影状态、收集命令和展示 facts。文字密集的周计划、事件、菜单、预测、账本、存档和开发者面板全部使用 DOM，不使用 Canvas。

### 5.4 持久化

浏览器本地保存版本化纯数据，至少包含 `schemaVersion`、`contentRevision`、PRNG 状态、命令序号和已决定的事件结果。自动保存发生在事务边界；提供一个手动槽和 JSON 诊断导出。

## 6. 最小状态

- 时钟：日、时段、剩余 AP、生活方针；
- 玩家：体力、上限、固定段位；
- 女主：体力、心情、好感、默契、固定的冷淡阶段；
- 酒馆：现金、人气、库存批次、设施、伙计解锁、当天计划；
- Aura：仅实现规格列出的少数实例；
- 故事：关系/调查机会与少量永久事实；
- 随机：可序列化 PRNG 状态；
- 审计：近期命令与结算 facts。

## 7. 错误处理与可解释性

- 内容引用在启动时验证，缺失 ID 阻止开始新局；
- 存档版本过新时拒绝加载并显示原因；
- 事务异常不得推进时间或扣除部分资源；
- 所有数值变化生成带 reason code 的 facts；
- 开发者面板能强制时段、事件和随机结果，并查看“为什么可用/不可用”；
- 不接外部遥测，诊断数据只保存在本地并由玩家主动导出。

## 8. 测试设计

- 单元测试：AP/体力守卫、FIFO 库存、腐败、菜单偏好、容量、营业账本、Aura 到期；
- 确定性测试：PRNG 固定向量、事件实例、同输入同结果；
- 存档测试：round-trip、损坏拒绝、最小 schema 迁移；
- 内容测试：稳定 ID、引用、条件与效果类型；
- Golden week：现金、关系、调查、全委托、两晚停业和显式失败策略；
- 薄 E2E：新游戏、第一天营业、保存、刷新、继续到周日。

## 9. 非目标

本规格不设计正式视觉风格、完整剧情、长期经济、多周目、Mod、Unity 架构、成人版本或商业发布。参考项目只提供通用工程启发，任何第三方实现或内容均不得复制。

## 10. 评审重点

评审时只需要确认：

1. 七日范围是否足以验证核心循环；
2. 周五互斥是否过于人为，还是适合作为实验控制；
3. 两种设施与四种营业方式是否覆盖想验证的取舍；
4. 数值草案是否可以作为第一轮假设，而不是正式承诺；
5. 规格通过后再编写实现计划，当前不继续扩系统。
