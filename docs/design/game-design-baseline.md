# 酒馆经营、关系养成与文字冒险游戏设计基线

日期：2026-07-10
状态：高层方向已确认；当前由七日 Web PoC 验证核心循环
正式生产候选：PC、Unity 6.x、URP
当前验证环境：React + TypeScript，可丢弃原型

## 1. 项目定位

这是一个个人长期 side project。首要目标是学习游戏开发并持续产出可玩的成果，不以短期发行或固定工期为前提。

游戏发生在日式低魔幻想世界，以酒馆经营 SLG 为主轴：玩家在有限时间、体力和金钱中经营酒馆，同时选择是否投入唯一女主的关系、城镇重建和战争往事。酒馆之外的行动以文字冒险表现，不进入实时战斗。

第一阶段是青少年级别的健全版本。未来可以把成人版本作为独立产品重新评估，但当前内容、架构和商店规划均不依赖它。

## 2. 设计支柱

### 2.1 每个回合必须有焦点

系统丰富度不能变成每日检查清单。玩家应当知道本时段、本日和本周主要在解决什么；常规设置可以沿用，重复劳动逐步由伙计、生活方针和已掌握流程代办。

### 2.2 取舍来自选择，不来自遗忘

玩家错过内容，应当是因为把有限资源投入了另一件有价值的事，而不是忘记点击隐藏按钮。会关闭路线的期限、条件和不可逆结果必须提前给出可理解的信号。

### 2.3 失败可解释且通常可恢复

普通经营失误带来小额损失、短期状态或恢复压力。重大失败必须经过预警和补救窗口。随机检定采用 fail-forward：失败改变代价和后续，而不是要求回到事件前重掷。

### 2.4 有限主轴相互交汇

- 酒馆与小镇重建：所有游玩共有的主轴；
- 战争真相：可以投入或放弃的副轴；
- 女主关系：横跨经营与故事的关系层。

镇民、旅人、商会和旧战友用于连接这些轴，而不是各自拥有互不相干的攻略管道。

### 2.5 关系表现本身是玩法

不同长期关系阶段、当前心情和临时状态改变同一互动的措辞、态度、可用选项与协作结果。关系玩法不是单纯把好感数值刷满。

## 3. 明确排除项

首个正式版本与当前 PoC 均不依赖以下内容：

- 实时地下城、实时战斗、敌人、装备和战利品构筑；
- 烹饪、钓鱼、QTE 等独立小游戏；
- 复杂员工排班、员工个人路线和深度员工养成；
- 自由装修沙盒；
- 运行时开放式 LLM 对话；
- 多女主、角色外貌自由定制；
- 第一/第三人称自由探索；
- 面向玩家的完整 Mod 编辑器和 Workshop；
- 成人内容。

## 4. 时间、行动与生活方针

每天分为上午、下午和晚上；凌晨只在方针、剧情或特殊状态下开放。行动点表示时段内可投入的时间，不模拟真实分钟。体力是独立约束，用来制造“有时间但撑不住”与“有体力但来不及”的差异。

休息消耗行动点并恢复体力；睡眠质量、设施和方针影响跨日恢复。天气、季节与复杂作息在 PoC 后再验证。

每周选择一次生活方针。方针调整一周的时间分布、恢复或自动化，不成为每日维护系统。候选包括均衡生活、夜猫子、认真工作和睡懒觉；当前 PoC 只比较其中两种。

## 5. 状态、Aura 与事实记录

显式核心状态使用专属字段：金钱、库存、属性段位、好感、默契、关系阶段、体力、心情和主路线状态不能被塞进万能 flag。

Aura 用于补充这些字段不能自然表达的短期处境或窄语义标记：

- 可见 Buff、Debuff 或中性状态；
- 有明确时长、解除条件或“无自动过期”；
- 隐藏剧情事实和一次性触发标记；
- 少量作用于某次营业或事件的修正。

关键长期记忆使用显式、可查询的记录；隐藏 Aura 不得静默造成重大路线失败。若一个内部 flag 会产生可规划的重大后果，必须通过对白、任务或预警投影给玩家。

典型例子：

- `生气`：临时覆盖互动和协作，不改变长期关系阶段；
- `备菜状态良好`：只影响下一次营业，随后消失；
- `获得战争线索`：永久事实，不显示成普通 Buff；
- `本周期缺货提示已触发`：周期内一次性标记。

## 6. 酒馆经营

核心循环：

`预测需求 → 采购 → 备菜 → 选择菜单与营业方式 → 结算销量、库存、现金、体力、人气和默契 → 支付固定成本并投资`

菜谱通过购买、剧情、探索和书籍获得。原料有保质期，成品可能报废；储藏设施缓解而不消除库存判断。

顾客根据菜单、价格、人气、客群和事件产生需求。缺货失去收入并可能小幅降低人气；计划停业与临时无法营业必须区分。普通失误不直接破坏长期关系。

### 6.1 员工自动化

伙计是工具角色。营业方式包括：

- 亲自与女主营业：现金和默契最好，消耗时间与体力；
- 部分委托：中等收益和消耗，释放少量时间；
- 完全委托：支付工资、容量较低，但释放整个营业时段；
- 计划停业：无收入，避免临时缺货或失约惩罚。

完全委托不要求在每一天都与亲自营业等价；它的价值是把时间换给关系、冒险和恢复。它不能成为无论怎样都不应选择的陷阱。

### 6.2 固定设施树

设施采用公开的固定升级树，不做自由装修。方向暂定：

- 厨房与菜谱；
- 大堂与接待；
- 储藏与自动化；
- 起居与记忆。

起居节点如舒适床铺、浴室、壁炉、书桌、地图台和遗物陈列，不必直接增加营业额。升级应优先解锁规则、事件解法或自动化；纯百分比加成只作补充。

## 7. 玩家属性与检定

玩家拥有体格、社交和智力，段位为 C、B、A、S、S+。底层经验值用于成长，界面和剧情主要使用段位。

段位用于：

- 解锁确定性事件选项；
- 为不确定检定提供加值；
- 参与路线、结局和特殊内容条件。

段位加值暂定 C=0、B=1、A=2、S=3、S+=4。

确定性门槛与随机检定分开：满足 `[智力 B]` 的选项直接成功；临场风险使用 `2D6 + 段位 + 准备 + 状态`。同一动作通常不在高门槛之后再次掷骰。随机结果由确定性 PRNG 产生并保存在状态中，读档不能刷新。

## 8. 女主关系

关系由以下维度组成：

- 默契：共同工作形成的协作能力；
- 好感：女主对玩家的个人评价；
- 关系阶段：通过明确事件转移的长期状态机；
- 关键记忆与边界记录；
- 女主体力与心情；
- 当前临时状态。

正向阶段为：

`陌生 → 冷淡 → 友好 → 信任 → 爱慕 → 恋人`

`厌恶`是关系危机旁支，不是好感数轴的最低一格。低好感加严重或反复的边界记录只会开放危机事件；危机处理失败或继续越界，女主才会离开并形成 Bad End。

不发展关系是完整合法路线。双方可以保持冷淡但高效的经营伙伴状态完成游戏。短期的`生气`不会静默降级关系，解除后恢复原有阶段。

## 9. 剧情结构

以下内容只冻结人物动机、序章功能与几条叙事轴，尚不是世界观或主线剧本定稿。正式内容阶段仍需单独完成背景设定、章节结构、角色小传和事件表。

### 9.1 序章

主角曾是勇者，在敌国陷阱中重伤，仅与一名战友幸存。两人被经营小镇酒馆的老人救下；战友离开追查真相，主角因伤留下成为帮厨，老人的养女担任服务生。

战争随后波及小镇，酒馆被毁。老人保护城镇后力竭而亡，临终留下遗物并托付养女，希望酒馆得以重建。老人的死亡是短小可玩序章的结尾。

### 9.2 网状叙事

章节拥有共享关键事件。局部选择改变事实、台词、解法和结局权重，并在一到两个事件内重新汇合。普通事件限制读取的上下文，避免条件组合爆炸；日常场景采用共享正文加关系差分，只有里程碑和重大冲突制作完整独立场景。

追查战争真相与专注生活不是善恶选择。执着复仇可以通向逐渐失去新生活的 Bad End，但不建设为完整的恶人玩法。

## 10. 文字冒险

森林、遗迹、边境和任务地点使用文字冒险。出发前选择准备、同行者、路线与投入时间；途中通过门槛、2D6 和资源选择解决问题。

产出是稀有食材、任务物品、情报、人物记忆和新事件入口，而不是装备刷取。失败可能造成受伤、延误、物资损失或不完整情报，并继续推进故事。

冒险必须回流经营，经营设施、资金与关系也必须改变冒险解法。

## 11. 结局与多周目

结局分为酒馆、女主关系、战争故事三个维度，再组合为后日谈。中性关系、放下往事和普通经营成功均是完整结局。

长期可以通过不同结局和关键经历获得有限元进度，换取每周目两到三项开局加成。金钱、关系、厌恶记录和整棵设施树不继承。圆满结局是多次经历后的综合挑战，不否定此前结局。

多周目、元进度与结局组合均不进入当前 PoC。

## 12. 重复内容与 Mod

后续周目可以跳过精确看过的对白变体，自动结算已经掌握的常规流程，并在出现新台词、状态、选项或检定时停止。跳过系统减少劳动，不替玩家做新战略选择。

第一阶段只要求内部数据驱动，不交付公开 Mod 工具。内容只能使用代码拥有的有限条件与效果类型，不执行任意脚本。稳定 ID 不复用，重命名通过 alias/tombstone 迁移。

## 13. LLM 边界

LLM 优先用于开发期的草案、变体、测试样例和一致性检查；正式内容经人工选择、修改并落盘。主要游戏循环不依赖网络生成。

## 14. 技术与美术方向

当前 Web PoC 用纯 TypeScript 模拟核心和 React 灰盒 UI 验证玩法。验证通过后，只迁移规则规格、数据定义、确定性测试向量和验收场景到 Unity/C#，不直接移植 TypeScript 运行时。

正式生产候选为 Unity 6.x + URP。视觉方向用 3D 场景和角色保证一致性，并允许固定镜头内容预渲染成 2D。正式美术、卡通渲染、换装、自由镜头和资产生成管线不得成为玩法验证前置条件。

当前个人技术偏好是进入 Unity 阶段时从 6.5 起步，并在 6.7 适合项目后再评估升级；CoreCLR 迁移是希望顺带体验的技术方向，不是玩法、架构或进度的前置条件。真正开工时仍以当时稳定性、插件兼容性和迁移成本重新确认版本。

个人项目当前可接受的总预算上限约为 10,000 美元，但它是风险容忍上限，不是必须花完的制作目标。支出按“先验证风险、再购买对应资产或插件”分阶段发生，不用采购替代尚未成立的玩法。

正式制作可以组合商业成品素材、商业插件和 AI 生成素材。每项外部资产都要记录来源、版本、许可证、是否允许修改与商业发布；除非许可证明确允许，不把购买素材输入 AIGC 服务，也不把参考项目的代码、文本或资源用于生成管线。资产风格统一与 AIGC 管线在玩法 PoC 后单独验证。

## 15. 首轮验证问题

1. 玩家能否立即说出今日与本周目标？
2. 经营、关系和调查是否形成真实取舍？
3. 普通遗漏是否只产生可解释的小后果？
4. 亲自、部分委托、完全委托和计划停业是否都有具体用途？
5. 不发展关系时，冷淡伙伴路线是否成立？
6. 门槛、准备、设施与关系是否提供不同解法？
7. 状态、心情、好感、默契和关系阶段是否容易区分？
8. 营业结算能否解释每一笔收入、损失和变化？
9. 同一存档、输入与种子是否完全可复现？
10. 重复一周后，哪些操作已经疲劳，应该合并或自动化？

这些问题由 `docs/poc/` 下的七日 PoC 回答；当前数值不是正式承诺。
