# Goal Execution Protocol v1

状态：当前有效。Contract ID：`project-tavern.goal-execution.v1`。

本协议约束 Phase 0 与 Phase 2–6 的所有 agentic execution。它把原来依赖外部 workflow skill 的隐式做法变成仓库内自包含合同。

## 1. 工具中立性与责任

- 不要求安装任何外部 workflow skill/plugin；仓库内协议已经完整定义执行纪律。
- Goal、计划更新、子代理、代码审查代理和浏览器工具都只是可选执行手段。缺失、容量不足或委派失败时，主执行者必须回退到本地实施和自审，不能因此阻塞。
- 主执行者必须亲自读取 `GOAL.md`、Roadmap、当前 phase plan 和当前 task 的权威输入，对最终范围、修改、验证、暂存和提交负责；不能用子代理摘要代替这些责任。
- 子代理只接收有边界、可独立验证的实现或审查子任务。并行结果必须由主执行者在 live tree 上复核后才能进入 task commit。

## 2. 执行与恢复证据

执行与恢复证据只来自：

1. Git ancestry 与 task commit；
2. 默认 phase checkpoint：最后一个 accepted task commit、当前通过的 phase acceptance gate 与 clean worktree status；
3. phase plan 明确要求的 tracked checkpoint 或 materialization/visual 等本机 attestation；
4. 当前只读 focused/phase/full gate；
5. 当前完整 worktree 状态。

Phase plan 的复选框保持 `[ ]`，仅表示有序验收步骤。执行时不把它们改成 `[x]`，也不另建聊天专用完成表。模型记忆、旧对话和未验证摘要都不能覆盖 live evidence。

## 3. 单任务状态机

每次只允许一个 task 处于实现中：

1. **定位：**确认当前 phase base、最后 accepted task commit、`HEAD`、worktree 和当前 task `Files`。
2. **输入：**读取 task、直接权威规格、现有 public interfaces 和相关 tests；确认没有设计冲突。
3. **RED：**先加入最小聚焦测试，并运行计划指定命令。只有命中指定 test title、缺失 symbol/path 或稳定 diagnostic code 才是合格 expected-red。
4. **GREEN：**实现满足权威合同的最小闭集，不提前实现下一 task，不顺手重构范围外代码。
5. **聚焦验证：**运行 task 的 focused tests、类型/构建和计划指定 checkpoint。
6. **差异审查：**检查完整 tracked/untracked set、`git diff --check`、架构/许可/边界和生成 bytes。可使用独立审查代理，但主执行者必须复核结论。
7. **精确暂存：**只暂存 `Files` 与计划明确列出的 rename/delete/generated set，比较 `git diff --cached --name-status`；禁止无范围的 `git add -A`、amend 和混入用户改动。计划明确给出带 pathspec 的 `git add -A -- <paths...>` 时，该闭集操作允许。
8. **提交：**focused 与 task gate 通过后创建计划规定的 task commit。commit 是长 Goal 的恢复点，不 push、不部署。
9. **提交后复验：**检查新 SHA、clean/预期 status、关键 gate 和不得变化的 lock/baseline；然后自动进入下一 task。

意外的语法、依赖、环境、浏览器、端口或无关测试失败不能冒充 expected-red。先诊断并恢复到能证明目标缺口的 RED。

## 4. 恢复算法

首次运行、continuation 或中断后：

1. 读取 `git status --short --branch`、最近五个 commits、最近默认或显式 checkpoint 和当前 phase gate。
2. 若 clean tree 且最后 task commit/验证匹配，从下一 task 开始，不重做已验收工作。
3. 若 dirty tree 的全部路径都属于同一个未完成 task，重建该 task 最近一次可证明的 RED/GREEN，随后从第一个未证明步骤继续。
4. 若 dirty tree 跨 task、包含范围外用户文件、来源不明的 generated bytes 或与 checkpoint 不一致，保留现场，不 reset、不覆盖；输出路径分类和所缺决策。
5. writer 中断时，在临时目录重新生成候选并逐字节比较。相同 bytes 可复用；foreign bytes 必须停止，不能自动覆盖。
6. 已存在但无法由 commit + gate 证明的实现不自动视为完成；先做只读验证和窄 diff 审查。

模型容量错误、进程退出、Goal continuation、缺失聊天上下文或没有可用 subagent，都按本算法恢复，不从 Phase 起点重来。

## 5. 修复与停止分类

| 分类                                                                     | 处理                                                                               |
| ------------------------------------------------------------------------ | ---------------------------------------------------------------------------------- |
| 当前 task 内普通实现/测试缺陷                                            | 修复并继续                                                                         |
| 已验收 earlier owner 的实现偏离，但规格答案唯一                          | 回到最早 owner，创建窄 repair commit，重跑从该 owner 到当前的受影响 gate，然后继续 |
| 计划命令、路径或 allowlist 与 live tree 机械不一致，且唯一修复不改变规格 | 在同一 repair commit 修正文档/脚本并验证；记录原因后继续                           |
| 权威规格互相冲突或需要改变公共合同/产品行为                              | 停止并请求设计决定                                                                 |
| 新外部依赖、未物化浏览器/字体、凭据、网络或权限                          | Phase 0 内按 stable code 诊断；主 Goal 中停止，不临时下载                          |
| 素材采用、许可判断、人工试玩或远端发布                                   | 移交对应独立轨道，不在主 Goal 内执行                                               |
| 范围外 dirty work、foreign baseline bytes、无法确认来源的 state          | 保留现场并停止                                                                     |
| workflow skill/plugin/subagent 不可用                                    | 本地执行和自审；继续                                                               |

同一外部阻塞应在计划允许的安全检查和替代方案耗尽后才报告。不能仅因为任务复杂、耗时或需要较多上下文而标记 blocked。

## 6. Baseline 与 writer

- 普通 `verify`、focused verifier、phase gate 和 release gate全部只读，不接受 tracked byte 漂移。
- fixture、golden、command corpus、materialization lock 和 screenshot 只能由计划指定 writer 生成。
- writer 先在临时目录形成完整候选，暴露 untracked 文件，记录排序后的 path/size/SHA-256，再按 task rubric 审查。
- 工程 fixture/golden/screenshot 可由执行 agent 按 schema、identity、diff 和视觉 rubric 审查，无需例行人工暂停；这不构成素材批准或主观试玩。
- 任何更新 baseline 的 task 都必须同时提交 writer、review evidence 和只读 verifier，并证明普通 gate 不调用 writer。

## 7. Checkpoint 合同

默认 phase checkpoint 是一个可从 live tree 重建的验收边界，由以下三项共同构成：

- phase 最后一个 accepted task commit（同时是 checkpoint `HEAD`）；
- 当前重新通过的 phase acceptance gate；
- clean worktree status，或 phase plan 明确允许的预期状态。

默认 checkpoint 不新增 tracked 文件、不创建独立 commit。恢复时重新运行 gate；若它与 Git ancestry 或 worktree 不一致，以 live evidence 为准并调查漂移。

只有 phase plan 在 task `Files`、写入步骤、精确暂存集合和 commit message 中明确要求时，才创建 tracked checkpoint。该显式文件至少记录：

- `goalBaseCommit`、`phaseBaseCommit`、`lastAcceptedTaskCommit` 和 checkpoint `HEAD`；
- phase/task ID、执行命令、退出结果和关键稳定 diagnostics；
- public ABI、fixture/golden/source graph/visual/artifact 等适用摘要；
- materialization digest 与适用的本机环境 fingerprint；
- 完整 worktree status、任何保留的用户变更及其未暂存状态；
- 下一 phase plan 和第一个未执行 task。

Checkpoint 是证据，不是第二套运行时状态。执行者不得仅为方便恢复而自行发明计划外证据文件。

## 8. Commit 与外部副作用授权

采用本协议启动 Phase 0 或主 Goal，即授权执行计划内每个通过验收的窄 task commit；只有 phase plan 明确给出完整文件与提交合同时才授权对应 checkpoint commit。不授权 push、PR、发布、部署、远端配置、资产批准或计划外依赖安装。

主执行者必须保留所有范围外用户改动。若它们使 clean-boundary task 无法安全开始，报告 exact paths，而不是 reset、stash、覆盖或偷偷包含进 task commit。

## 9. Phase 与 Goal 完成

- Task 完成：task commit 存在，focused/task gate 通过，staged scope 正确，提交后状态满足计划。
- Phase 完成：所有 task 均满足上条，phase acceptance 通过，并在最后一个 task commit 上形成默认 checkpoint；若计划另有显式 checkpoint 合同，还必须满足该合同。
- Goal 完成：Phase 2–6、Roadmap Definition of Done、release evidence 和最终工作树报告全部完成。

Phase completion、生成一个可打开页面、单个 E2E 通过或预算接近上限都不是 Goal completion。

## 10. 协议场景验收

| 场景                                       | 必须行为                                                |
| ------------------------------------------ | ------------------------------------------------------- |
| main Goal 在素材 handoff 或 Phase 0 前启动 | 不编辑 Phase 2；报告 exact missing precondition         |
| clean tree 已有匹配 task commit            | 复验后跳到下一 task                                     |
| dirty tree 全在当前 task allowlist         | 重建证据并原地恢复                                      |
| dirty tree 含范围外用户改动                | 保留并分类；不能暂存或覆盖                              |
| expected-red 因无关 syntax/环境失败        | 不计为 RED，先修复意外失败                              |
| writer 中断但候选逐字节相同                | 复用并继续审查                                          |
| writer 中断且存在 foreign bytes            | 停止，不覆盖                                            |
| earlier owner 出现规格答案唯一的回归       | 窄 repair commit，重跑受影响链后继续                    |
| 需要资产/设计/远端权限决定                 | 停止并移交对应责任人或轨道                              |
| 外部 skill 或 subagent 不可用              | 本地执行和自审，继续                                    |
| 模型/进程中断后 continuation               | 从 commit/checkpoint/live status 恢复，不重做整个 phase |
